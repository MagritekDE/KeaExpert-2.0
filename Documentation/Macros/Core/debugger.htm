Macro Debugger

 

 

The debugger macro allows other Prospa macros to be debugged i.e. single stepping through the macro command by command, applying break points and viewing variables.

 

 

The menu is almost identical to the usual text editor menu found in Prospa. Start by using the open macro command in the file menu to load the macro you wish to debug into the editor window. Once the macro is displayed you can start execution by pressing the "Start debugging" button. Once this button has been pressed you have several restrictions on what you can and cannot do:

 

  1. The rest of Prospa is no longer accessible, however you can move windows around on the screen.
  2. Menus in the debugger are no longer accessible.
  3. The text in the debug editor can no longer be edited but you can add break points by left clicking on the break point margin. This added a solid red circle. Clicking on this again will remove it.
  4. The output window can be used like a normal command line interface, however now you also have access to window and local variables. (In the normal CLI you only have access to global variables).
  5. The "Variables to list" text box can contain any variables which you wish to display at each break point – the most common might be "local" which will display all local variables or "winvar" which will display all window variables.
  6. The current command to be executed is indicated by the yellow arrow. Which commands are then executed is controlled by the 5 buttons at the top left:

·         Step over. This runs the next command in the current procedure but will not enter a procedure command. The procedure is called and returned in one step.

·         Step in. This runs the next command in the current procedure but if this command is a procedure call the debugger will start stepping through the commands in that procedure. The new procedure will be displayed in the editor window.

·         Step out. If you are in a procedure and press this button it will run all commands automatically until it has left that procedure. The parent procedure will then be displayed.

·         Run to break. This will run all subsequent commands until it has reached a break point (selected by left clicking in the break point margin). When the break point is reached execution will pause and any variables requested will be displayed.

·         Stop debugging. This will run all subsequent commands in normal, non-debugging, mode. This should be the button you press to finish a debug session.

 

Some other points to note:

 

  1. You can't close a window which is being debugged – stop debugging first.
  2. Break points can be added to procedures in other files and will be remembered for the lifetime of the debugger window.
