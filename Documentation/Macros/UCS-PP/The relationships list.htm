The relationships list

 

 

The relationships list in the pulse program provides a connection between the pulse program pulse amplitudes and delays with the “NMR” parameters exposed in the user interface (e.g.  90/180 pulse amplitudes and echo-times).

 

This is often the most difficult part of the pulse program to write, as determining the various delays can be quite complex. The difficulty arises because in the NMR experiment we would like events to occur a certain times. However in the pulse program language there are no absolute times, just delays between events. Because the spectrometer computer is a serial device, there are a number of small delays which must be included to ensure correct timing. The result is a relatively complex expression which writes each delay in terms of the NMR parameters.

 

An example of this is given below for the spin-echo sequence:

 

 

Here the relevant part of the relationships table takes the following form.

 

   relationships = [...

                    "a1 = 90Amplitude",

                    "a2 = 180Amplitude",

                    "d1 = pulseLength",

                    "d2 = echoTime/2-pulseLength-pgo",

                    "d3 = echoTime/2-acqTime*500-pulseLength/2+rxLat",

                    ...]

 

On the left side we have the pulse program parameters and on the right side the NMR parameters. Notice that we have to consider the pulse gate overhead (pgo) delay (typically 1.5 us – this is defined in the pulse command) when calculating d2, and in d3 we need to include the receiver latency parameter rxLat. (This takes into account the propagation delay through the digital receiver and also the 6 dwell times which are ignored at the start of the acquisition period – see the acquire command). Also note that the acquisition time is in milliseconds so acqTime*500 is just half the acquisition time in microseconds.

 

Tables

 

Tables can also be included in the relationships table. For example to generate a soft RF pulse:

 

   relationships = [...

                    "ns = nrSteps",

                    "d2 = stepLength",

                    "s = sin(linvec(-1,1,ns)*2*pi+1e-8)/(linvec(-1,1,ns)*2*pi+1e-8)",

                    "n1 = size(s)",

                    "t1 = keaRun:convertTxGain(spAmp)*mag(s)",

                    "t2 = 32768*(s <= 0)",

                    ..."]

 

...

 

   shapedrf(mode,t1,t2,p2,n1,d2) 

 

Note that once a variable has been defined in the table (e.g. ns) it can be used in an expression further down in the list (e.g. the definition of table s uses ns). You can treat each string in the relationships table as a line of code in a normal Prospa macro.

 

In this example a shaped RF pulse is generated using table t1, which is a sinc function (table s) scaled by the user interface variable spAmp (in dB). t2 is the phase table, which just contains zeros for positive values of s and 32768 (= 180 degrees) for negative values of s.

 

Functions

 

When the calculations required to perform the relationships table are too complex to fit on one line you can call a procedure. In the following example from the spin-echo imaging macro

 

   relationships = ["readGrad = 2*pi*nrPnts/(acqTime*1e-3*gamma*FOV*1e-3)",

                    "phaseGrad = acqTime*500*readGrad/(echoTime/2-500)",

                   "(n1,n2,n3,n4,n5,n6,n8) =   seimaging:setImagingPlane(plane,readGrad,

                                                 phaseGrad,xshim,yshim,zshim,xcal,ycal,zcal)",

                   ...]

 

readGrad and phaseGrad are intermediate results which are sent to the procedure setImagingPlane in the seimaging macro. Some of the variables in the argument list are global to the experiment (such as xshim, ... xcal, ... see below) while others are defined in the user interface list (plane).

 

The returned parameters (n1, n2 ...) are used to set the various gradient values in the experiment. Note that the procedure call should include the name of the parent macro (since it is evaluated in keaCtrl.mac not just in seimaging.mac) and it must always return something – even if it is a dummy parameter –  since the list is evaluated using the Prospa assignlist command which assumes each entry in the list is assigned to something.

 

Global parameters

 

Apart from the user interface variables all the parameters in the Kea preferences dialog can accessed along with some derived parameters defined in keaCtrl:inialisePP. The most important ones are listed below:

 

autoScale ............ will data be autoscaled in the 1D plots (yes/no)

calibrateForPreamp ... should the preamplifier be included in the data calibration (yes/no)

calibrateForRxAmp .... should the receiver amplifier be included in the data calibration (yes/no)

dataSymbolColor ...... RGB color for data symbols

dataSymbolShape ...... Shape for data symbols

fitColor ............. RGB color for fit curves

freqMax .............. Maximum allowable frequency in plots (autoScale = no)

freqMin .............. Minimum allowable frequency in plots (autoScale = no)

gradCtrlMaxOP ........ Maximum output voltage from the gradient controller.

gradCtrlRes .......... The number of bits in the gradient controller DAC.

hpaLocation .......... The location of the RF amplifier (int/ext)

keepShimsOn .......... Will the gradient shims be kept on between experiments (yes/no)

maxDutyCycle ......... The maximum allowable RF duty cycle for an experiment.

maxFrequency ......... Maximum allowable frequency in experiments.

minFrequency ......... Minimum allowable frequency in experiments.

oampcal .............. Other gradient amplifier calibration (A/V)

ocoilcal ............. Other gradient coil calibration (T/A/m)

oshim ................ Other gradient shim values (T/A/m)

pgo .................. Pulse gate overhead (us)

preampGain ........... Preamplifier gain

rxAmpCorrection ...... Receiver amplifier gain correction

rxAmpOffset .......... Low frequency receiver amplifier offset correction

rxCalibration ........ Transceiver receiver calibration (/uV)

rxLat ................ Receiver latency (us)

spectrumColor ........ RGB color of relaxation and diffusion spectral traces

timeMax .............. Maximum time in plots (autoScale = no)

timeMin .............. Minimum time in plots (autoScale = no)

units ................ Plot time axis units (uV/mV/raw)

useCommonPar ......... Use common parameters between experiments (yes/no)

warnOnOverwrite ...... Warn if files are being overwritten (yes/no)

xampcal .............. x gradient amplifier calibration (A/V)

xcoilcal ............. x gradient coil calibration (T/A/m)

xshim ................ x gradient shim values (T/A/m)

yampcal .............. y gradient amplifier calibration (A/V)

ycoilcal ............. y gradient coil calibration (T/A/m)

yshim ................ y gradient shim values (T/A/m)

zampcal .............. z gradient amplifier calibration (A/V)

zcoilcal ............. z gradient coil calibration (T/A/m)

zshim ................ z gradient shim values (T/A/m)
