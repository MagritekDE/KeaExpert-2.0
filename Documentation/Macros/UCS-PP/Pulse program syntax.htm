Pulse program argument syntax

 

This section defines the syntax of the pulse program commands argument. Note you will need to add the Prospa DLL “kea2PPRun.dll” to the Prospa DLLs folder before the kea commands will be recognised. It will also need to be the correct version (2.16 is the latest version at the time this manual was written).

 

To find the individual command syntax please refer to the help page for that command.

 

Like all Prospa commands, each pulse-sequence command takes several arguments. However most of these arguments are not the parameter values but rather the parameter names. Previous versions of the pulse programmer required that these be quoted strings, however from V1.09 of the DLL onwards this is no longer necessary. You can now also enter constants for a number of the parameters. If they are variables then the parameter values are sent to these commands via the pulse program parameter list.

 

Example for the T1-IR-Add macro (before V1.09)

 

# Pulse sequence

   initpp(dir) # Reset internal parameter list

 

   cleardata("n2")

   pulse(mode,"a2","p1","d1")     # 180 RF pulse

   wait("w1")                     # 180 - 90 delay

   pulse(mode,"a1","p2","d1")     # 90 RF pulse

   delay("d3")                    # 90 - 180 delay

 

   loop("l1","n1")                # CPMG loop

      pulse(mode,"a2","p3","d1")  # 180 RF pulse

      delay("d4")                 # 180 - acq delay

      acquire("sum","n2","d5")    # Acquire echo and wait

   endloop("l1")

 

   lst = endpp() # Return parameter list

 

New syntax from V1.09

 

# Pulse sequence

   initpp(dir) # Reset internal parameter list

 

   cleardata(n2)

   pulse(mode,a2,p1,d1)     # 180 RF pulse

   wait(w1)                 # 180 - 90 delay

   pulse(mode,a1,p2,d1)     # 90 RF pulse

   delay(d3)                # 90 - 180 delay

 

   loop(l1,n1)              # CPMG loop

      pulse(mode,a2,p3,d1)  # 180 RF pulse

      delay(d4)             # 180 - acq delay

      acquire("sum",n2,d5)  # Acquire echo and wait

   endloop(l1)

 

   lst = endpp() # Return parameter list

 

If you use the second form for parameter entry then be careful not to define a variable with any of the above names. (They are only interpreted as parameter names because there are no existing variables defined with these names).

 

Command variables

 

There are several types of parameter or variable used by the Kea commands. These variables are listed below – in all cases they consist of a single letter followed by any valid string. In the standard macros this has been an integer greater than zero (e.g. n2, a12, b65). In the following text “externally” refers to how the variable appears to the end user while “internally” is how the variable is stored on the spectrometer processor. From version V1.09 it is also possible, in many cases, to pass a constant instead of a parameter.

 

ax …   an RF amplitude – externally represented by a decibel value ranging from -85 dB (= 3.2 pW) to 0 dB (= 1 mW). Internally the amplitude is represented by a 14 bit number. If you pass this as a constant it has a range from 0 to 16383.

 

bx …   a byte value use to set the TTL output. Externally represented by an 8 bit number (0-255). Internally it is the top 8 bits of a 16 bit number. The lower 8 bits control internal TTL lines. If you pass this as a constant then the allowed values are hex integers in the range 0x01 … 0x80. (1-128 decimal).

 

dx …   a delay in ms. Externally this can range from 2 to 327670 ms. Internally it is stored as a 24 bit number which is used to control a hardware timer. Constant values can be from 0.25 to 327670 ms.

 

fx …    a frequency in MHz. Externally this is a floating point number ranging from 0 to 400 MHz. Internally it is represented by two 24 bit numbers (i.e. a 48 bit word).

 

gx …   the receiver amplifier gain setting. Externally represented by a decibel value ranging from -20 to 70 dB in 3 dB steps. Internally it is two 8 bit number each controlling the gain of one RF stage. (For the low frequency Kea the numbers range from 0 to 60 dB externally and internally by a 16 bit number.)

 

lx …    a loop index – this string is used explicitly to label DSP code loops – it is not an external parameter.

 

nx …   an integer. Internally this is stored as a signed 24 bit number however the valid external values will depend on the particular command.

 

px …   an RF phase shift relative to the receiver phase. Externally this can take values 0, 1, 2, 3 corresponding to 0, 90, 180, 270 degrees. Internally the phase is represented by a 16 bit number. If you pass this as a constant it has a range from 0 to 65535.

 

tx ....    a row vector containing a table of values, e.g. linvec(-1,1,100). The range of values will depend on the output being controlling.

 

wx …  a long delay in ms. Externally and as a passed constant this can range from 2 ms to 167 s. Internally it is stored as three 24 bit numbers.

 

A frequently asked question is why we don’t allow arbitrary variable names as in the rest of Prospa. The answer is that there is usually a conversion factor between external and internal units and the first letter identifies the data type, thereby allowing this conversion to be applied automatically. It also makes the type of variable immediately identifiable to the reader. However as noted above there is no reason why the rest of the variable name can’t be more descriptive than a number – this has been done for historical reasons. The exceptions to this are the phase parameters which should have numeric suffixes so their position in the phase table can be determined.
