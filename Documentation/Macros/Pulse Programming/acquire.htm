acquire

 

This command collects the specified number of complex data points from the digital receiver. This information can be manipulated in several different ways depending on the mode parameter.

 

Syntax 1

 

acquire(mode, points, [[duration]])

 

mode ................... one of:

 

overwrite :     each data point is stored in a new location in memory. If the command is run more than once the second data set will overwrite the first.

 

append:         each data point is stored in a new location in memory. If the command is run more than once the subsequent data sets will be appended to the last collected.

 

integrate:      all points collected are added together and the result stored in the next memory location.

 

 

points ................. the number of complex points to collect. Has the form "nx".

 

duration2 ............. this optional parameter defines the total duration of the command. If the required acquisition is shorter than this, then a delay will be added to the end. If this option is not included, then no delay will be added. Has the form "dx".  Beware of the case when the acquisition time is actually longer than the duration. (See note 2).

 

addressOffset …... DSP only. Where data should be stored. This is an offset from the base data address y:0x10000. If this is not supplied, it defaults to the current data address which may be incremented by previous commands. Note that only the append and integrate acquire modes modify the current data address. However, the memreset command can also be used to change the base address. The address offset variable has the form "mx". Note that the offset is for complex data words.

 

Syntax 2

 

acquire(mode, points, duration, [shift])

 

mode ................... one of:

 

sum/stack:   each data point is stored in a new location in memory. Subsequent data sets will be added to the first to improve SNR. (Make sure you use the cleardata command at the start of the pulse program for this mode).

 

points ................. the number of complex points to collect. Has the form "nx".

 

duration2 ............. This defines the total duration of the command. If the required acquisition is shorter than this, then a delay will be added to the end. If this option is not included, then no delay will be added. Has the form "dx".  Beware of the case when the acquisition time is actually longer than the duration. (See note 2).

 

shift ………………...    To prevent the stacked data from overflowing the internal registers it can be scaled before summing. Here shift is a power of 2 which is used to divide down the data by shifting it to the right by this number of steps. The default shift is 0.

 

Syntax 3

 

acquire(mode, points, [shift, [[duration]])

 

mode ................... one of:

 

integrateandscale:    all points collected are added together after a power of 2 scaling and the result stored in the next memory location.

 

 

points ................. the number of complex points to collect. Has the form "nx".

 

shift ………………...    To prevent the stacked data from overflowing the internal registers it can be scaled before summing. Here shift is a power of 2 which is used to divide down the data by shifting it to the right by this number of steps. The default shift is 0.

 

duration2 ............. this optional parameter defines the total duration of the command. If the required acquisition is shorter than this, then a delay will be added to the end. If this option is not included, then no delay will be added. Has the form "dx".  Beware of the case when the acquisition time is actually longer than the duration. (See note 2).

 

 

Notes

 

  1. Because a digital filter is applied to the data as it is collected, there will be a filter dependent delays between sampling the RF data and it being stored in memory. This is called the receiver latency and can be incorporated into the relationships list using the predefined variable rxLat. The receiver latency takes the following values

 

   if(flatFilter == "yes")

      rxLat = 1.73 – 0*DW

   else

      rxLat = 1.73 - 3*DW

   endif

 

where DW is the dwell time in ms. The possibility of a negative latency is explained below. For more details on the digitization of the NMR signal see the section Acquisition Explained in the Programming Guide.

 

 

  1. The acquisition duration is longer than would be predicted by the formula dwell_time×number_of_points. This is because the total number of points collected is always larger by 6. These points are then discarded because they are distorted by the digital filter as the receiver switches on. The receiver latency variable rxLat takes this loss of points into account so that you will still start acquiring data at the correct time. In addition, there is time at the end of the acquisition period for data to be transferred to the spectrometer memory. This time depends on the number of points collected, the dwell time plus a number of internal parameters. You can determine the total acquisition time within your execpp procedure by calling the following function:

 

 acqTm = ucsNMR:getAcqTime(guipar)

 

3.  Note that the acquisition command is blocking. i.e. you can’t do anything else while this is happening. The alternative is to use the acquireon and acquireoff commands which are nonblocking.
