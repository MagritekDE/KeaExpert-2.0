2.                     User definable interface

 

In Prospa 2 the main windows were predefined. The Prospa window held a series of semi-user defined menus, 1D, 2D and 3D plot windows (1 each only), a single text editor and command line interface. By dividing some of these windows it was possible to display multiple plots or text files but in a way which was less than standard for most Windows users.

 

 

In Prospa 3 the core windows are still available but they are now user definable using more fundamental windows objects such as windows, texteditors, menus etc. This means that you can not only have multiple plot windows or text editors but you can also define a completely different user interface from the standard one supplied. 6.                     Multiple threads

 

Previous versions of Prospa could only do one thing at a time.  V3.1 allows multiple threads to be run providing the illusion that more than one macro is being run at the same time.  An example macro demonstrating this is shown below.

 

procedure(test1dthread)

 

  pd = curplot("1d")

  thread(":core",pd)

 

endproc()

 

procedure(core,pd)

 

   while(1)

   for(k = 10 to 1000)

      pd->plot(noise(k))

      pause(0.01)

   next(k)

   for(k = 1000 to 10 step -1)

      pd->plot(noise(k))

      pause(0.01)

   next(k)

   endwhile()

 

endproc()

 

This thread can be aborted by pressing the escape key.

 

 

7.                     New controls

 

To make the Prospa interface more user-definable, addition graphical user interface controls have been added including

 

1.      1, 2 and 3D plots.

2.      Command line interface(s) although normally you would only have one.

3.      Text editors.

5.      The updown control for incrementing or decrementing a number.

6.      Pictures, for displaying a jpeg or other type of graphics file in a window.

7.      Tabs for controlling which objects will be visible in a window. (Previously done with buttons).

8.      Toolbars for defining an alternative interface to menus.

9.      Menus for defining your own window menus.

 

 

8.                     Better help

 

Prospa now has a built in HTML help viewer which means that adding help for new commands is as simple as writing an html file and putting it in the appropriate documentation folder. Previously help files had to be compiled using HTMLHelp which made it difficult for other people to add help for their macros or DLL commands.

9.                     Multiple plot and editor windows

 

Previous versions of Prospa only allowed a single 1D/2D/3D Plot and a single editor window. Although some of these windows could be divided into subplots or subeditor windows, it was not possible to display data or macros in several windows. With the introduction of the new plot and editor commands multiple plots can appear in a single window or each window can have an individual plot control (examples are provided in the layout menu). This provides Prospa with a more conventional windows interface. Note that individual plots and editors can still be subdivided as previously.

 

 

10.                Copy/paste facilities between plot windows

 

Along with the introduction of multiple plot controls comes the ability to cut, copy and paste plots between windows using the normal Ctrl+X/C/V key sequence. Copying a plot in this way also copies the plot to the clip-board as a windows meta file (.emf) which can then be pasted into another application. Holding down the shift key while copying will copy all subplots in a plot control. In a 1D plot holding down the shift key while pasting adds the trace to the current plot without removing other traces or resetting the plot labels.

 

 

11.                Macro debugger

 

Previously, debugging macros required placing print statements at appropriate locations and then running the code. Version 3 introduces a prototype debugger (in the gui menu) which allows single stepping through the macro code, adding and running to break points and interrogating variables interactively. The debugger will work on macros with and without a window interface. Note that the debugger will not work with multithreaded code.

Limited support for four dimensional data sets is now included.

 

 

13.                Double precision support

 

Prospa previously only worked with single precision floating point numbers and matrices. Most of the time this is sufficient; however for some algorithms (e.g. the calculation of standard deviations) this may lead, in certain cases, to considerable error. For these special cases new commands such as single, double and dmatrix have been added to allow the manipulation of data with full double precision  that means 15 significant figures rather than 7.

 

m = double(n) ...       converts the single precision scalar or 2D matrix 'n' to a double precision scalar or 2D matrix m

n = single(m) ...        converts the double precision scalar or 2D matrix 'm' to the single precision scale or 2D matrix n.

d = dmatrix(n,m) ...  generates a zeroed double precision matrix d with m rows and n columns.

 

Adding a 'd' to the end of a number will make it double precision. e.g. 3.1415927d

 

 

14.                 Improved designer features

 

An improved window GUI designer is now provided which in addition to all previous functions has support for the addition controls Prospa 3 includes and extra buttons for attaching controls to the right or bottom part of a window so when it resizes the control will move with the window. It also includes grid and snap-to-grid options to allow easier placement of controls.

 

 

 

15.                Class functions for windows and controls

 

With the introduction of plot controls the issue arises of how to access data in different plot controls. The Prospa 2.x method was to define a current plot and then access plot parameters and data using a number of commands (such as plot, image, axes, grid ...) which work on the current plot data. Although it would be possible to continue accessing all plot data by using the current plot concept (a new command would be required to set the current plot to a different control) an alternative method for those cases where modifying the current plot is undesirable or inconvenient has been introduced. This is to add a command which returns the plot object to the user. This object allows access and modification to all parameters within the plot without changing the current plot. One case where it is essential that the current plot is not changed is when running multiple threads where one thread is updating one plot and another thread is allowing interactive access to another plot.  An example of this would be when running an NMR experiment in the background in one window while allowing the operator to interact with another window.

 

Taking this idea further Prospa 3 provides object support for all windows and controls providing name based alternatives to the setpar and getpar commands. e.g.

 

amplitudeCtrl->text(123.4)

 

sets the text in the textbox called amplitudeCtrl to 123.4 while

 

value = amplitudeCtrl->value()

 

returns its value. The handle or identity (ID) of the control "amplitudeCtrl" can be obtained when the control is formed, by defining an objectID using the setpar() command or by extracting the object directly using the getobj() command.

 

All available class functions can be found in the Prospa help viewer under the class command list.

 

16.                Structures supported

 

In version 2.x variables could be collected together using parameter lists. However the limitation was that each variable should be relatively small so that each element of the list could be represented as a string on a single line e.g.

 

lst = ["name = \"Joe\"", "age = 23", "address = \"18 Small St\""]

 

However for large data sets or objects not supported by the list commands (such as control handles) the structure provides a more useful and conventional method of storing and manipulating such data. e.g.

 

a = struct()

a->name = "Joe"

a->age = 23

a->address = "18 Small St"

a->map = sin([0:0.01:10])'*(cos([0:0.01:10]))

 

If the last entry were non-deterministic (like a real map) or very large, it would be impossible to store using the list method.

 

These structure elements may be used directly or converted into local variables using the assignstruct command.

 

procedure(structtest)

 

   a = struct()

   a->name = "Joe"

   a->age = 23

   a->address = "18 Small Street"

   a->map = sin([0:0.01:10])'*(cos([0:0.01:10]))

   assignstruct(a,"local")

 

   pr local

 

   image(map)

 

endproc()

18.                New submatrix array referencing

 

Submatrix access can now be achieved using array within array syntax:

 

m = [1:0.5:20]

s = m[1:5]

 

This extracts elements 1, 2, 3, 4 and 5 from matrix m and places them in matrix s.

 

Access from the start of a matrix to a certain index or from a specified index to the end can be achieved using the following syntax:

 

m = [1:10]

s = m[:3]

s = m[3:]

 

This method can also be used to extract all elements from one or more dimensions:

 

m = noise(4,4)

n = m[:,2]

 

It can also be used with the step size operator:

 

m = [0:10]

n = m[4:2:]

 

This last example extracts elements 4, 6, 8 and 10 from matrix m.

 

Negative number can be used to refer to matrix elements relative to the end of the array. -1 is the last element -2 the second to last and so on. In this way you don't need to know the size of the matrix.

 

> v = [1:10]

> pr v[-2]

 

   v[-2] = 9
