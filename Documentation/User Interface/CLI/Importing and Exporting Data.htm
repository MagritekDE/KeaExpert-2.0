Importing and Exporting Data

 

 

Using Prospa commands to import and export data

 

Prospa commands can also be used to import and export data in a variety of formats. Because of this, the command has been broken into two parts. First the format is specified using the command importxdpar or exportxdpar where x is one of 1, 2 or 3. These commands take pairs of strings - the first being the parameter (e.g. "ab") and the second being the value for this parameter (e.g. "ascii"). This format provides the possibility of easy extension in the future should a new file type appear which is not covered by the current parameter list. Once the parameters have been chosen the import or export commands are then run - these have a similar format to the load and save commands.

 

Example

 

To import a 2D k-space data set "test" with 128 rows, each row being made of 256 complex data points in big-endian, 32 bit floating point format, one would use the following command sequence:

 

import2dpar("ab","binary","machine","bigend",

            "xyrc","complex","fls","float")

m = import2d("test",256,128)

 

Big-endian machines (such as PowerPCs and SPARC) store data with the most significant byte first, while little-endian machines (such as PCs and other computers using Intel based processors) store data with the least significant byte first).

 

Example

 

To export a 1D complex vector, "v",  to a file "out.txt"  in ascii (text) format you would use the following commands:

 

export1dpar("ab","ascii","xyrc","complex")

export1d(v,"out.txt")

 

Using the menu interface to import and export files

 

Because of the number of formats available, the simplest way to accomplish import and export is to use the commands which appear in the Files menu of the main Prospa window.

 

When importing data you must supply all the necessary parameters. The first step is to select the dimension. Then the appropriate import dialog will appear. For example for a 1D selection we obtain:

 

 

 

 

Note that the filename extension has no significance to the import procedure. To import the data you will need to know whether it is in ASCII (text) format or binary format and whether it is real or complex. ASCII data can be delimited with spaces, commas or tabs. The import routine will work this out itself. Likewise you do not need to specify the number of data points if the file represents 1D data. Complex data is assumed to consist of real-imaginary pairs placed one after the other. Again the delimiters used are not important, although they should not be mixed. (If they are you will get an error message like: "Odd number of values in data set".) However you can place linefeeds wherever you like in the data set to make it more readable since these are ignored. And example of a complex or x-y data set is given here:

 

1,1

2,2

3,9

4,16

5,25

6,36

7,49

8,64

9,81

10,100

 

The "Y data" option treats each number as a y-data value, while the "XY data" option expects (x,y) data pairs (i.e. two data sets will be loaded).

 

If you choose the binary option you will also need to choose the byte order and the binary format. If the data comes from a non-Intel based machines (e.g. old Power PC based Macintoshes) then case check the "big-endian" box. There are also 3 binary formats supported: 32 bit floating point, 32 bit integer and 16 bit integer. You will need to refer to the documentation from the data source to determine which type is appropriate. (Although you can make some educated guesses based on the size of the data set).

 

 

Like the load menu command you can choose the names of the variable(s) the data should be associated with and whether the data should be displayed or not.

 

When importing 2D or 3D data there is an extra set of text fields allowing the dimensions of the data set to be included as well (see above).

Examples:

 

Try importing the following files from the import folder. The relevant parameters are:

 

phased_spectrum.ascii:     1D complex, ASCII file

kiwi:                                  2D, 256 by 256,  real, binary, little-endian, 32 bit float

kumquat.txt:                      2D, 256 by 256, real, ASCII.

small.3d:                            3D, 95 by 58 by 30, real, binary, big-endian, 32 bit float.

 

Any error messages which appear during the loading probably mean that you have made a mistake with the parameters, since the program did not read all the data or expected more than there was. (Although sometimes you may just wish to read a part of the data set - perhaps the first plane in a 3D file for example). Some data sets have a binary header at the beginning which contains information about the data. In Prospa we can skip this by entering the header size, in bytes, into the import dialog.
